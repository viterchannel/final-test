## [STARTING] T1 — Reproduce the rider login bug
Inspected the live `platform_settings` table:

| key | value |
|---|---|
| auth_phone_otp_enabled | `{"customer":"on","rider":"on","vendor":"on"}` |
| auth_email_otp_enabled | `{"customer":"on","rider":"on","vendor":"on"}` |
| auth_username_password_enabled | `{"customer":"on","rider":"on","vendor":"on"}` |
| auth_magic_link_enabled | `{"customer":"on","rider":"on","vendor":"on"}` |
| auth_google_enabled | `{"customer":"off","rider":"off","vendor":"off"}` |
| auth_facebook_enabled | `{"customer":"off","rider":"off","vendor":"off"}` |
| auth_2fa_enabled / biometric | `{"customer":"off","rider":"off","vendor":"off"}` |

So the toggles are seeded correctly with per-role JSON. The bug
must therefore arise either when (a) the row is missing entirely
on a fresh install, or (b) the wrong cache is read after an admin
toggles something off.

## [COMPLETED] T1 — root causes identified
Four defects, each independently capable of producing
"No login methods available":

### Bug A — Settings cache desynchronisation
There were **two independent in-memory settings caches** in the
API server:

| File | Cache vars | Importers |
|---|---|---|
| `routes/admin-shared.ts` | `_settingsCache`, `_settingsCacheExpiry` | most `routes/admin/*.ts` save endpoints |
| `middleware/security.ts` | `settingsCache`, `settingsCacheExpiry` | `routes/auth.ts`, rate-limit middleware |

Calling `invalidateSettingsCache()` from `admin-shared` did **not**
clear the security-side cache — they're separate variables. After
an admin toggled a switch, the auth router served stale values for
up to 30 s. Worst case, if security.ts cached an empty `{}` after
a transient DB error and then the row was added, the rider
endpoints stayed broken until the process restarted.

### Bug B — `parseAuthToggle` ignored its `_fallback` parameter
`routes/platform-config.ts` parses every per-role auth toggle for
the public `/platform-config` response. When the DB row was
missing (fresh install or partial seed), the helper returned plain
`false` instead of the documented default (`"on"` for phone OTP,
email OTP, username/password). Every client (rider, vendor,
customer) then showed the empty-state error.

### Bug C — Rider login error was too generic
`Login.tsx` collapsed every `no_method` reply into the same
sentence, so users couldn't tell whether the platform was
misconfigured, their account was disabled, or the network call
had failed.

### Bug D — Audit/refresh-token tables lacked supporting indexes
`auth_audit_log`, `refresh_tokens`, `pending_otps`, and
`magic_link_tokens` had no index on the columns the admin queries
filter on (`user_id`, `event`, `ip`, `created_at`, `expires_at`).
Audit-log paging in the Admin Panel scanned the whole table.

## [STARTING] T7b — Per-role registration kill-switches
The Admin Panel exposes "Vendor Registration" and "Rider Registration"
toggles (`vendor_registration`, `rider_registration` in
`platform_settings`), but the API never read them — only
`feature_new_users` (a global) was honoured. An admin who
disabled vendor signups expected new vendor accounts to be
blocked while customer signups stayed open.
## [COMPLETED] T7b
Added per-role guards in `routes/auth.ts` `/auth/register`
(after the global `feature_new_users` check):

```ts
if (userRole === "vendor" && (settings["vendor_registration"] ?? "on") === "off") {
  res.status(403).json({ error: "Vendor registration is currently closed by the administrator." });
  return;
}
if (userRole === "rider"  && (settings["rider_registration"]  ?? "on") === "off") {
  res.status(403).json({ error: "Rider registration is currently closed by the administrator." });
  return;
}
```

Defaults to `"on"` so unset rows keep working. The toggles use
the unified settings cache, so admin saves take effect on the
next request without a process restart.
